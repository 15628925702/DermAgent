# 一、你一共要设计哪些内容

你最终其实要设计 5 个核心部分。

## 1. 主体诊断流程

也就是整个 DermAgent 怎么从输入走到最终诊断。

这个部分回答的是：

病人图像和metadata进来以后
先做什么
再做什么
什么时候调用skill
什么时候去查经验库
什么时候停止
最后怎么输出

这个是你的**主干架构**。

------

## 2. Skill 库

这个部分回答的是：

系统一共有多少种 skill
每个 skill 干什么
什么时候调用
调用后返回什么

这个是你的**能力系统**。

------

## 3. Experience 库

这个部分回答的是：

系统记住的到底是什么
是病例？经验？规则？混淆对？失败教训？
怎么存
怎么检索
怎么压缩
怎么写回

这个是你的**长期记忆系统**。

------

## 4. Planner / Router

这个部分回答的是：

谁来决定调用哪个 skill
是固定流程还是动态组合
为什么这个病例要调 compare skill
为什么另一个病例要调 malignancy risk skill

这个是你的**agent中控层**，也是你和普通pipeline最大的区别。

------

## 5. Learning / Writeback 机制

这个部分回答的是：

每做完一个病例，系统怎样积累新经验
哪些经验值得保存
哪些经验要压缩
哪些错误以后要避免
skill 要不要更新提示词/规则/模板

这个是你的**continual learning 机制**，也是最像 XSKILL 的地方。

# 三、逐块讲解这个架构图

------

## Phase A：Experience & Skill Accumulation

这部分不是给新病例直接做诊断，而是在构建你的“底座”。

它主要包含三块：

### A. General Perception Skill

这一块是通用感知层。

输入是：
皮肤图像 + 可用的metadata + 用户问题

输出不是最终诊断，而是一些**粗粒度中间结果**，比如：

视觉线索
粗粒度类别
top-k 初始候选
初步风险提示
初步DDX

比如一个病例进来，感知层输出：

边界略不规则
有角化
颜色不均
初步怀疑 ACK / SCC / SEK
存在一定恶性风险，需要进一步确认

这一层的意义是：
它先把问题从“完全未知”缩小到“几个候选病种范围”。

所以它是后面 planner 做决策的前提。

------

### B. Experience Generation

这一块是经验生成层。

它的输入不只是当前病例，而是你已有的大量历史结果，包括：

过去病例
混淆病例
错误案例
不同skill使用后的效果
诊断推理过程
最终标签和反馈

它要做的事情不是单纯存原始病例，而是抽取出更高层次的经验条目。

比如从很多病例里抽出：

ACK 和 SCC 容易混淆
如果“角化明显 + 隆起更强 + 恶性风险更高”，更偏向 SCC
如果“表浅 + 粗糙鳞屑 + 老年暴露部位”，更偏向 ACK

再比如：

MEL 和 NEV 的混淆边界
BCC 和 SEK 的对比经验
某类病例只靠 perception 不够，必须加 compare skill

这一层其实就是把“病例”提升成“经验”。

------

### C. Experience & Refinement

这一块是经验整理与压缩层。

它做两件事。

第一件事是把经验规范化，变成统一结构。
第二件事是压缩。

为什么要压缩？

因为如果你把每个病例都原样塞进 memory，以后会越来越乱，检索成本会越来越高，而且会有大量重复经验。

所以你需要把经验分层：

原始病例经验
类别原型经验
混淆对经验
可复用规则经验

举个例子。

原始经验：
这个病例是 ACK，图像有鳞屑、粗糙、日光暴露部位，曾和 SCC 混淆。

压缩后经验：
对于 ACK/SCC 混淆，若以表浅粗糙角化为主且浸润证据弱，优先考虑 ACK；若隆起、浸润、恶性风险更明显，优先考虑 SCC。

这就是你师兄说的：
**不是简单病例库，而是经验库，并且经验要有选择性压缩。**

------

## 图中间：Skill Library + Experience Manager

这块是整个系统的“资源中心”。

### Skill Library

这里放的是你可以随时调用的 skill。

不是所有 skill 每次都用，而是按需调用。

### Experience Manager

这里负责从经验库中做检索和筛选。

检索标准不是随便搜，而是结合：
相似度
数量控制
质量筛选
可用性判断

也就是说，不是把所有历史都丢给模型，而是给它最有帮助的 top-k 经验。

------

## 图中间下方：per-case shared state

这块特别重要。

你可以把它理解成每个病例的一块“共享黑板”。

所有模块都往里面写信息，比如：

初始候选
置信度
检索经验
对比结果
恶性风险评估
已调用skill列表
最终汇总证据

它的意义是：
整个 agent 不是每个 skill 各干各的，而是共享同一个病例状态。

你现在代码里的 `case_state` 就是这个东西的雏形，后面继续升级就行。

------

## Phase B：Dynamic Skill Routing & Diagnosis

这部分才是新病例真正的诊断阶段。

它主要分三块。

------

### E. Experience-Skill Driven Planner

这是核心中的核心。

这一层决定：

这个病例需不需要比较 skill
需不需要恶性检查 skill
需不需要调用某个专病 specialist
检索哪些经验
是否已经足够确定可以停止

也就是说，它不是固定流程，而是动态路由。

比如：

如果 top1 和 top2 差距很小
就调 compare skill

如果候选里有 MEL / BCC / SCC
就调 malignancy risk skill

如果 ACK 和 SCC 混淆明显
就调 ack_specialist_skill 和 scc_specialist_skill

如果 retrieval 支持很强、风险很低、候选一致
那就少调一些 skill，直接出结果

所以 planner 是系统真正像 agent 的地方。

------

### Dynamic Skill Selection

这一块是 planner 的执行体现。

它根据当前病例状态，选择接下来要走的路径。

注意，这里不是单选，而是可以组合。

一个病例可能走：

perception
retrieval
compare
malignancy risk
mel specialist
final decision

另一个病例可能只走：

perception
retrieval
final decision

还有一个病例可能走：

perception
retrieval
compare
ack specialist
scc specialist
final decision

这就是你师兄说的：
**skill 灵活选择组合。**

------

### Dynamic Specialist Skills

这就是专病 skill 层。

它不是通用粗看，而是针对特定病种或混淆对做深入判断。

比如：

MEL specialist
重点检查色素不均、边界不规则、恶性信号

ACK specialist
重点检查表浅角化、粗糙鳞屑、日光暴露部位先验

SCC specialist
重点检查浸润感、恶性风险、角化隆起程度

BCC specialist
重点检查珍珠样边缘、结节感、局部特征

Compare skill
专门比较两个候选谁更合理

Malignancy risk skill
不直接定病种，而是判断恶性倾向，给最终决策施加约束

这层就是你的“精细诊断武器库”。

------

### Reflection & Writeback

这一块是闭环学习。

病例做完以后，系统不只是吐一个结果结束，而是要回写经验。

它会写回什么？

这个病例最后诊断是什么
调用了哪些 skill
哪些 skill 有帮助
哪些地方出现了混淆
最终形成了什么可复用经验
有没有必要生成新的压缩经验

所以系统会越来越强，不是一次性的。

这就是你论文里最像 continual learning 的部分。

# 五、你最终的 Skill 库怎么设计

你的 skill 库不要设计得太杂，要分层，分成三类最合适。

------

## 第一类：通用技能

这些 skill 几乎所有病例都可能用到。

### 1. perception_skill

作用：
从图像和metadata中提取初始线索，给出粗粒度候选和风险提示

输出：
视觉线索
coarse category
top-k ddx
uncertainty
risk cues

------

### 2. retrieval_skill

作用：
从经验库里检索相似经验、混淆经验、原型经验

输出：
top-k experience items
support diseases
historical confusion hints

------

### 3. compare_skill

作用：
比较两个或多个候选病种谁更符合当前证据

输出：
pairwise comparison result
which candidate is better supported
key distinguishing evidence

------

### 4. malignancy_risk_skill

作用：
评估病例是否存在恶性倾向

输出：
benign / suspicious / malignant-like 风险等级
风险证据

这个 skill 很重要，因为它可以作为 safety skill，不让系统轻易把高风险病错判成低风险病。

------

### 5. metadata_consistency_skill

作用：
检查视觉特征与 metadata 是否一致

比如：
年龄、部位、病史是否支持当前候选

输出：
consistency score
conflict notes

------

### 6. uncertainty_assessment_skill

作用：
判断当前证据是否足够，是否还需要进一步 specialist

输出：
uncertainty level
need_more_skills = yes/no

------

### 7. report_skill

作用：
生成最终诊断报告

输出：
final diagnosis
top-k ddx
evidence summary
risk notes
suggested follow-up

------

## 第二类：专病 specialist 技能

这些 skill 不是每次都用，只在可疑时调用。

### 1. ack_specialist_skill

重点判断 ACK 相关特征，尤其和 SCC、SEK 的边界

### 2. scc_specialist_skill

重点判断 SCC 的侵袭性和恶性指征

### 3. mel_specialist_skill

重点判断 MEL 的色素、边界、恶性倾向

### 4. nev_specialist_skill

重点判断 NEV 与 MEL 的差别

### 5. bcc_specialist_skill

重点判断 BCC 的结构性线索

### 6. sek_specialist_skill

重点判断 SEK 的良性表观与伪装性

------

## 第三类：策略型技能

这是最像 agent 的 skill。

### 1. confusion_pair_skill

输入两个候选病种
专门调取这对混淆病的历史经验和判别规则

### 2. escalation_skill

当高风险且不确定时，决定继续调用 specialist 或给出保守输出

### 3. reflection_skill

病例结束后生成“本病例经验总结”

### 4. compression_skill

把多个原始经验压缩成一条更抽象的经验

------

# 六、你最终的经验库怎么设计

这是你系统的另一条主线。

经验库不是一个表，而应该是分层设计。

------

## 第一层：Raw Case Experience

这是最原始的病例经验。

每条经验记录：

case_id
image embedding 或 image path
metadata
初始 perception 输出
最终标签
最终预测
top-k 候选
调用过的 skills
哪些证据支持最终决策
是否曾发生混淆
最终总结

这是“最细粒度记忆”。

------

## 第二层：Prototype Experience

这是类别原型经验。

它不是具体病例，而是某一类病的典型模式。

比如：

ACK prototype
MEL prototype
BCC prototype

每个 prototype 可以包含：

典型视觉特征
典型 metadata 模式
常见误判对象
推荐 skill

这层的作用是：
给 retrieval 更稳定的高层支持，而不是只靠零散病例。

------

## 第三层：Confusion Experience

这是最重要的一层之一。

专门存混淆对经验。

比如：

ACK vs SCC
MEL vs NEV
BCC vs SEK

每条经验里写：

为什么容易混
怎么区分
哪些线索最关键
建议调用哪些 specialist

这层和 compare skill 是天然配套的。

------

## 第四层：Rule / Insight Experience

这一层是压缩后的可复用决策规则。

比如：

当 top2 为 MEL 和 NEV，且恶性风险 skill 高，必须调用 mel_specialist_skill
当 ACK/SCC 混淆且 retrieval 支持弱，不要直接输出 ACK
当 benign 外观与高风险 metadata 冲突时，启动 metadata_consistency_skill

这层非常适合写成“经验压缩结果”。

------

## 第五层：Failure-Correction Experience

这层专门记录错例和纠正经验。

比如：

为什么之前错了
是 perception 不足
还是 retrieval 误导
还是少调了某个 specialist
以后遇到类似病例怎么修正

这一层很像“失败教训库”，对论文很加分。

------

# 七、经验库怎么检索

你不要只做一种检索，而要做两阶段检索。

## 第一阶段：粗检索

根据图像特征、metadata、coarse category 去找相关经验

目的：
快速筛出候选经验池

------

## 第二阶段：精筛选

根据当前候选病种和不确定性去筛更有用的经验

比如：
如果当前是 ACK/SCC 混淆
那优先取 ACK/SCC confusion memory
而不是泛泛的所有病例

所以你最终检索出来的经验不该只有“最像的病例”，而应该是：

相似病例经验
原型经验
混淆对经验
可复用规则经验

这才是“经验库”而不是“病例近邻库”。

------

# 八、你最终的 Agent 怎么设计

你的 agent 建议设计成 6 个模块。

------

## 1. Input Parser

负责接收：
图像
metadata
问题描述

转成统一 case_state

------

## 2. General Perception Module

先做粗粒度判断，生成初始候选和线索

------

## 3. Planner / Router

根据当前状态决定下一步调用什么

这是 agent 的大脑。

它的输入是：
perception 输出
retrieval 支持
uncertainty
risk level
历史经验

它的输出是：
selected_skills

------

## 4. Skill Executor

真正按 planner 的要求执行 skill

------

## 5. Decision Aggregator

把多个 skill 的结果合并成最终判断

注意这里不是简单投票，而是要做证据聚合：

哪个skill支持哪一类
哪个skill是高优先级 safety skill
是否存在恶性约束
是否还有冲突没解决

------

## 6. Reflection & Memory Writer

病例结束后回写经验库，并决定是否压缩/更新

------

# 九、你整个 agent 的执行流程应该是什么样

一条完整流程可以写成这样：

输入病例
→ perception_skill
→ retrieval_skill
→ planner 分析当前 uncertainty 和 risk
→ 若候选接近，调用 compare_skill
→ 若候选含恶性风险，调用 malignancy_risk_skill
→ 若存在专病可疑，调用对应 specialist skill
→ decision aggregator 汇总所有证据
→ report_skill 输出最终报告
→ reflection_skill 生成病例经验
→ writeback 到 experience bank
→ compression_skill 定期压缩经验

这就是你系统最完整的闭环。

# 十、你的论文创新应该怎么表述

你最终不要把创新说成“我用了很多skill”，而要说成三层创新。

## 创新一：经验库不是简单病例库

提出分层经验记忆机制
同时保存原始病例经验、原型经验、混淆对经验和压缩规则经验

## 创新二：动态 skill 路由

不是固定pipeline，而是根据候选、风险和不确定性动态选择 specialist 技能组合

## 创新三：对比式专病检查

当系统怀疑某些相似病种时，先做候选对比，再触发对应专病检查 skill，完成 coarse-to-fine 诊断

你师兄说的三点，基本就被你完整落地了。
